## Backend & Mini App Roadmap (Durak TMA)

### 1. База проекта и инфраструктура
- **1.1 Структура Go-сервиса**
  - Уточнить финальный layout пакетов (`/cmd/api`, `/internal/{auth,users,wallet,payments,matchmaking,game,admin,i18n}`, `/pkg`).
  - Зафиксировать контракты между REST и WS слоями (общий error envelope, trace-id, локализация).
- **1.2 Конфигурация и окружения**
  - Описать `.env` и config-структуру (prod/stage/local; Telegram bot token, DB/Redis, провайдеры платежей).
  - Поднять `docker-compose` для PostgreSQL + Redis + API + миграции (расширить уже существующий стек).
- **1.3 Миграции и схемы**
  - Развернуть миграции для таблиц: `users`, `wallets`, `transactions`, `games`, `game_players`, `game_events`, `payouts`, `admin_logs`, `bans`.
  - Добавить индексы по ключевым полям (user_id, game_id, created_at, status).

### 2. Авторизация и сессии (Telegram Mini App)
- **2.1 Валидация initData**
  - Реализовать модуль валидации HMAC-SHA256 initData по токену бота.
  - Добавить middleware, извлекающее `user_id`, `username`, `locale` из initData.
- **2.2 User lifecycle**
  - Авто-создание пользователя в БД при первом заходе Mini App.
  - Хранение статуса (active/banned), локали, базовых метаданных Telegram.

### 3. Кошелёк и финансы
- **3.1 Модель кошелька**
  - Имплементировать `wallet` с полями `balance_usd`, `locked_usd` и транзакции по типам (deposit/withdraw/bet_lock/bet_release/payout/fee).
  - Гарантировать инварианты: `balance_usd >= 0`, `locked_usd >= 0`, `balance_usd + locked_usd` согласованы с историей транзакций.
- **3.2 Платёжные провайдеры (абстракция)**
  - Описать интерфейсы `CryptoProvider` и `CardPSP` (инициация платежа, статусы, вебхуки).
  - Добавить слой идемпотентности по внешнему `provider_op_id`.
- **3.3 REST и вебхуки**
  - `POST /wallet/deposit/init`, `POST /wallet/withdraw` с валидацией лимитов и AML-политики (упрощённый вариант).
  - `POST /payments/crypto/webhook`, `POST /payments/cards/webhook` с полным логированием курса, комиссий и статуса.

### 4. Матчмейкинг и столы
- **4.1 Создание и поиск столов**
  - Реализовать сущность стола (`games` + `game_players`) с настройками: колода, режим (подкидной/переводной), fair/shuler, ставка, число игроков.
  - Эндпоинты `POST /tables/create`, `POST /tables/join`, `POST /tables/ready`.
- **4.2 Подтверждение ставок**
  - `POST /tables/confirm-stake`: блокировка ставки (escrow) для всех готовых игроков.
  - Обработка таймаута подтверждения и отмена игры с разблокировкой средств.

### 5. Игровой движок “Дурак”
- **5.1 Машина состояний**
  - Имплементировать FSM: `Lobby → Deal → RoundStart → AttackPhase ↔ DefensePhase → Cleanup → RefillHands → NextRound → Finish`.
  - Хранить снэпшот состояния партии в Redis (для быстрого доступа) и писать события в `game_events` (для реплея).
- **5.2 Правила и валидация**
  - Валидация атак, защите, подкидываний, переводов с учётом лимитов стола, колоды и статуса игроков.
  - Учесть особенности: лимит 6 карт, не больше карт защитника, поведение после пустой колоды.
- **5.3 Таймеры и AFK**
  - Таймер 25 сек на ход; `autoTake` для защитника и `autoEndTurn` для атакующего при таймауте.
  - AFK-логика при дисконнекте (>N секунд): маркировка, возможная бот-замена/пропуск хода (как отдельный флаг).
- **5.4 Режим “Шулер”**
  - Серверная реализация `shulerPlay` (разрешение “некорректного” удара) и `shulerReport` (окно 3 сек, деактивация способности для игрока до конца матча).
  - Логирование всех активаций и репортов для админки.

### 6. WebSocket-шина
- **6.1 Протокол событий**
  - Описать типы событий: join/leave, start, playAttack, playDefend, throwIn, translate, take, pass, endTurn, shulerPlay, shulerReport, timers, finish.
  - Ввести обёртку события с типом, payload, correlation-id и локалью.
- **6.2 Реконнект и восстановление**
  - При реконнекте по game_id восстанавливать актуальное состояние стола из Redis + последних событий.
  - Обрабатывать конкурирующие действия через серверный порядок (seq/lock).

### 7. Расчёт мест и выплат
- **7.1 Ранжирование игроков**
  - Реализовать определение мест по порядку выхода, с поддержкой ничьих (одновременный выход).
  - Хранить `place` и `share_percent` в `payouts`.
- **7.2 Финальный расчёт**
  - Считать банк, удерживать 3% комиссии, применять проценты по таблице (2/3/4 игрока и ничьи).
  - Создавать транзакции `payout` и `fee`, обновлять балансы и блокировки.

### 8. Админ-панель (API-слой)
- **8.1 Управление пользователями**
  - Эндпоинты `/admin/users`, `/admin/ban`, `/admin/unban`, `/admin/balance/correct`.
  - Логирование всех админ-действий в `admin_logs`.
- **8.2 Мониторинг игр и финансов**
  - `/admin/games`, `/admin/transactions`, базовая статистика по “Шулеру”, ставкам и выручке.

### 9. Локализация RU/UA
- **9.1 Словари**
  - Ввести словари для RU/UA на бэке (errors, системные сообщения, статусы).
- **9.2 Интеграция с Mini App**
  - Отдавать локализованные тексты в ответах и WS-сообщениях на основе `locale` пользователя.

### 10. Тестирование и качество
- **10.1 Юнит-тесты**
  - Покрыть: валидацию ходов, перевод/подкидывание, лимиты, добор, “Шулер”.
- **10.2 Интеграционные и E2E**
  - Матчи 2/3/4 игроков, сценарии таймаутов и реконнекта, финансы (escrow, комиссии, ничьи).
- **10.3 Нагрузочные тесты**
  - WS-сессии 1k+ комнат, мониторинг латентности и стабильности.

### 11. Оценка сроков (черновой порядок)
- **Инфраструктура и авторизация**: 3–5 рабочих дней.
- **Кошелёк и платежи (без сложного AML)**: 7–10 дней.
- **Матчмейкинг и игровой движок (без полного UI)**: 10–15 дней.
- **WS-протокол и реконнект**: 5–7 дней.
- **Расчёт выплат и админ-слой**: 5–7 дней.
- **Тесты, полировка, нагрузка**: 7–10 дней.
> Итоговая длительность зависит от степени параллелизации и готовности внешних провайдеров.

### 12. Основные риски
- **Платёжные провайдеры**: задержки интеграции, изменения API, KYC/AML-требования.
- **Игровая логика**: спорные кейсы “Шулера” и ничьих, необходимость доработки правил по фидбеку.
- **Нагрузочные сценарии**: пики по WebSocket и платежам, необходимость тонкой настройки БД/Redis.
- **Telegram/WebApp ограничения**: обновления платформы, изменение требований к initData/безопасности.
